# Main-Game (Exploration) HUD Parity & Completeness Audit

> **Provenance.** Generated by the `main-hud-parity-audit` multi-agent workflow (2026-07-08):
> 10 catalog agents (7 sighted-HUD regions + 3 mod slices; **257 sighted items, 131 mod items**) → 2 cross-ref
> agents (completeness + parity) → 16 adversarial per-finding verifiers (default-to-refuted) → 1 synthesis.
> **15 findings confirmed, 1 refuted** (see appendix). Each verifier's corrections are folded into the finding
> text below and override the original candidate wording.
>
> **Status (2026-07-08, same day): ALL 15 findings implemented, compile-clean.** The 5 parity leaks are
> gated, the 9 smaller completeness gaps are closed, and #4 landed as the full WrathAccess-ported
> `FormationScreen`/`FormationField` (preset radio + WASD 2-D editing field with pick-up/place, Shift+WASD
> glide with member-crossing cues, Comma review + Slash plant, Alt+digit grab, restore/preserve/close).
> A 20-agent adversarial review of the implementation confirmed 8 distinct follow-up defects (worst: spoken
> ability hotkeys off-by-one when an overdrive ability sits in the bar; "Paused" chatter on every pausing
> UI window) — all fixed in the same change set.
>
> **Scope.** The main-game (surface) HUD during **real-time exploration** — everything a sighted player sees
> while walking around an area out of turn-based combat, out of dialogs, out of full-screen windows. Excluded:
> the turn-based combat HUD (see `docs/combat-hud-parity-audit.md`), space/starship combat and the system/warp
> maps (separate future plan), the full-screen service windows themselves (their HUD entry buttons and badges
> ARE in scope), and main menu / settings / loading screens (except the area-name transition banner).

## Executive summary

The mod is **broadly at parity** for real-time exploration: the large majority of the 257 sighted-HUD information items across the passive-narration, exploration-scanning, action-bar, party-portrait, overtip and notification surfaces already reach a blind player — the full game-log corpus via `LogTap`, the whole warning/toast strip (including `addToLog:false` toasts), barks, loading-screen banners and hints, level-up and weapon-set announcers, the visibility-correct scanner / tile-cursor / sonar stack, the complete action-bar decision readout, the service-window button strip, tutorial and deck-transition screens, and the on-demand K gauges. There are **10 verified completeness gaps** (2 high, 6 medium, 2 low). The two high-severity ones are structural: **quest/objective notification banners are never spoken** — no game-log thread exists for quests, so the otherwise-universal `LogTap` channel *cannot* carry them (the single highest-value passive-narration gap on the surface HUD) — and **area-exit destinations are never named**, so the Exits category the scanner already lists cannot disambiguate which exit leads where. There are **5 verified parity leaks** (1 high, 3 medium, 1 low), and the systemic pattern differs from the combat audit: the proxy/scanner lens correctly folds the game's whole sighted-visibility model (`IsVisibleForPlayer` → fog + stealth + invisibility + `Features.Hidden`), but the **self-built raw-grid channels bypass it** — the Quote-key inspect resolver (high: an ungated fog/stealth oracle that persistently force-reveals never-seen units), the tile-cursor occupant read, the room map, and the wall tones all gate (at most) on the *tile's* fog state while the game gates on the *unit's own* visibility — and the points-of-interest list skips the game's `ILocalMapMarker.IsVisible()` for every non-loot pin type. **All findings below were implemented the same day — see the status note above.**

## (1) Completeness gaps (Rule 1) — ranked most decision-critical first

### High severity

#### 1. Quest and objective notification banners are never spoken — quest updates are invisible during exploration
The "journal updated" banner (quest title, New/Completed/Failed/Updated/Postponed state, new-quest description, objective titles, addendum text, and the icon-only "clue" mark) reaches a sighted player on every quest event; a blind player gets nothing pushed at all.
- **Sighted reads it at:** `QuestNotificatorVM.cs:61-135` (`HandleQuestStarted/Completed/Failed/Updated/Postponed` at `:61-84`, the four objective handlers at `:100-118`, gates at `:86-92,120-135`, `ForbiddenQuestNotification` cutscene/loading/GroupChanger deferral at `:35-45`); views `QuestNotificationQuestView.cs:48-59` (New-state description, state marks, `GetQuestNotificationStateText`), `QuestNotificationObjectivesView.cs:57-91` (objective title + the icon-only `IsClue` mark at `:74-82`), `QuestNotificationAddendumView.cs:44-50` (addendum text); `QuestNotificatorBaseView.cs:161-171` (errand/Rumours/RumourAboutUs exclusions), `:158` (`Journal.NewQuest` sound); state wording incl. the Quest/Rumour/Order noun at `UIQuestNotificationTexts.cs:85-96`.
- **Mod status:** *missing* — **no game log thread exists for quests** (verified: `Kingmaker.UI.Models.Log.CombatLog_ThreadSystem.LogThreads.*` contains no quest/objective thread), so `LogTap` and the L review *structurally* cannot carry it; the mod has no `IQuestHandler`/`IQuestObjectiveHandler` subscriber and no quest-notification locale keys (only `JournalScreen` keys exist). The only audible cue is the `Journal.NewQuest` sound — and that plays for **quest-level** notifications only (`QuestNotificatorBaseView.cs:158`); objective-level notifications (`ObjectiveNotificationsAction`, `:186-194`) play **no sound at all**, so objective updates currently have zero audible cue. The mod's `JournalScreen` does voice the game's attention flag as ", updated" per quest plus completed/failed state (`RTAccess/Screens/JournalScreen.cs:170-175,241-242`), so an updated quest is *findable* by browsing the quest list — but that is a pull-only full-screen window with no passive cue, and the notification's per-event content (which objective changed, the new-quest description, addendum text, the clue mark) is never pushed.
- **Fix:** New `RTAccess/Accessibility/QuestEvents.cs` (pattern: `ConvictionEvents`) implementing `IQuestHandler` and `IQuestObjectiveHandler` via `EventBus.Subscribe` — or a Harmony postfix on `QuestNotificatorBaseView`'s show path to inherit every gate for free. Mirror the game's gates exactly: `quest.Blueprint.IsSilentQuestNotification(state)`, `objective.IsVisible` + `silentStart` + `quest.State != None` (`QuestNotificatorVM.cs:86-92,120-135`), the view's errand/rumour exclusions (`QuestNotificatorBaseView.cs:161-171`), and defer while `ForbiddenQuestNotification` (`:35-45`). Speak the game's own localized state words (`UIQuestNotificationTexts.cs:85-96` incl. the Quest/Rumour/Order noun), `quest.Title` / `objective.QuestName` + objective title, the Description for `State == New`, and add a spoken "clue" word from `QuestNotificationEntityVM.IsClue` (icon-only even for sighted players). Queued speech, never interrupting.

#### 2. Area-exit destinations are never named — exits announce as raw GameObject junk with no "where it leads"
The exit overtip's destination title is one of the few **persistent** text elements on the sighted exploration HUD, and it is THE datum for choosing among exits.
- **Sighted reads it at:** `OvertipTransitionVM.cs:48` (`Title` = `AreaTransition Settings.AreaEnterPoint.TooltipDescription` — a real `LocalizedString`, `BlueprintAreaEnterPoint.cs:23,29`); `OvertipTransitionView.cs:59-63` (title shown whenever non-empty, no hover gate); local-map twin `AreaTransitionPart.cs:30-46` (`OnSettingsDidSet`) — note it uses `AreaEnterPoint.Tooltip(Settings.TooltipIndex)`, which respects the per-exit `m_TooltipList`, whereas the overtip's `TooltipDescription` always returns the index-less `m_Tooltip`.
- **Mod status:** *partial* — the Exits scanner category lists transitions with distance/bearing, Sonar plays a "transition" tone, and I starts the transition, so the player does hear that a thing *is* an exit; but `InteractableDescriber.ResolveName` (`RTAccess/Accessibility/InteractableDescriber.cs:322-363`) has no `AreaTransitionPart` case and transitions carry no `InteractionPart` (the mod's own comments at `ProxyMapObject.cs:251-253`), so the name falls to the cleaned GameObject-name fallback (`:361`) — dev-string junk. A blind player browsing three exits cannot tell which leads to the bridge vs the cargo bay without walking through each one and sitting through a load. Notably, the mod already contains the exact wanted readout, **dead**: `InteractableDescriber.DescribeMarker` (`:65-75`, doc example "Cargo hold, exit, 15 tiles, north") reads `marker.GetDescription()` — the very string `AreaTransitionPart` copies from the destination tooltip — but it has zero callers, and `Scanner.MarkerList` (`Scanner.cs:496-511`) deliberately excludes `LocalMapMarkType.Exit` markers, so that path never fires. (`TransitionScreen.cs` names destinations only for the separate multi-entrance fast-travel window, not walk-up surface exits.)
- **Fix:** Add an `AreaTransitionPart` case to `InteractableDescriber.ResolveName` (and hence `ProxyMapObject.Name`): prefer `AreaEnterPoint.Tooltip(Settings.TooltipIndex)` for exact per-exit fidelity when several exits share an enter point — or route through the object's `LocalMapMarkerPart.GetDescription()` instead of re-deriving from `AreaTransition.Settings` — falling back to the localized exit word when empty. No extra knowledge gate needed: the proxy's existing `IsInGame + IsRevealed + IsAwarenessCheckPassed` gate is at least as strict as the overtip's `IsVisibleForPlayer` (= `IsInGame` only, `OvertipTransitionVM.cs:53`). Also append it in the V room-exit cycle line when the opening coincides with a transition object.

## (2) Parity leaks (Rule 2) — ranked by severity

> **Status: all five leaks FIXED (2026-07-08).** The shared root cause for L1/L2/L4/L5 was that the
> self-built raw-grid channels bypassed the visibility model the mod's own proxy/scanner lens already mirrors
> correctly (`IsPlayerFaction || IsVisibleForPlayer`, which folds fog + stealth + invisibility + `Features.Hidden`
> via `View.IsVisible`); each now gates on that lens (or the fog probe, for the geometry channels).

### High severity

#### L1. Quote-key cursor inspect force-reveals and reads full stats of units the sighted player cannot see (fog / stealth / scripted-hidden)
- **The leak:** `RTAccess/Exploration/Inspect.cs:39` (`InspectCursor() => Run(MapCursor.Node?.GetUnit())`) resolves the tile cursor's occupant via the raw grid occupancy cache — `CustomGridNodeController.GetUnit` (`:44-49`), whose `Tick` (`:70-100`) indexes **all** awake units with no visibility filter — and `Run` (`:44-56`) gates only on `InspectUnitsHelper.IsInspectAllow` (`InspectUnitsHelper.cs:13-24` — faction/combat only, never visibility) before raising `IUnitClickUIHandler` and reading the full inspect sheet (wounds, defenses, weapons, abilities) via `TooltipReader`. The tile READOUT on the same never-seen tile correctly says only "unexplored" (`RTAccess/Accessibility/InteractableDescriber.cs:92-102`, `:100`), but Quote answers anyway. Worse: the inspect template ctor calls `InspectUnitsManager.ForceRevealUnitInfo` (`TooltipTemplateUnitInspect.cs:63`), so the press **persistently force-reveals** a unit the player has never seen and pops the visual panel. A blind player can sweep the cursor tile-by-tile through unexplored fog pressing Quote as a unit-detection oracle, and can read stealthed ambushers (`PartUnitStealth`, unspotted) or scripted `Features.Hidden` NPCs standing on lit tiles.
- **What the game masks:** `EntityVisibilityForPlayerController.cs:73-108` forces the unit view invisible for `Features.Hidden`, `IsHiddenBecauseDead`, `IsInvisible`, `IsInFogOfWar`, and Stealth-unspotted — and `EntityViewBase.SetVisible(false)` also **disables the layer-10 clickable colliders** (`EntityViewBase.cs:329-339`; units inherit `IsSelectableInFogOfWar => false` at `:128`), so a sighted player physically cannot click/hover/overtip such a unit. Every sighted inspect entry passes an already-visible unit, and `InGameInspectVM.OnUnitInvoke` has no visibility backstop, so the mod's raise goes straight through. The "opening Inspect force-reveals, so no gating needed" rationale in CLAUDE.md/the catalog **inverts cause and effect** — the game only force-reveals units the sighted player could already see and click.
- **Fix — gate to add:** In `Inspect.Run`, before raising `IUnitClickUIHandler`, require the same lens `ProxyUnit` uses: `unit.IsPlayerFaction || unit.IsVisibleForPlayer` (folds fog + stealth + invisibility + `Features.Hidden` via `View.IsVisible`; the mod's established pattern at `ProxyUnit.cs:46`, `InGameScreen.cs:506`, `UnitBuffer.cs:41`, `HitPredictor.cs:79`). On failure speak the existing localized `inspect.cant` / `inspect.none` refusal. The Y half (`InspectReview`) needs no change — `Scanner.SelectedUnit()` already comes through the visibility-gated browse. Note: this fix does **not** cover the tile readout's own occupant naming — that is leak L2 below and needs its own gate.

### Medium severity

#### L2. Tile-cursor readout names stealthed / invisible / scripted-hidden units standing on visible tiles
- **The leak:** `RTAccess/Accessibility/InteractableDescriber.cs:102` — `var unit = seen && !hideUnits ? node.GetUnit() : null` — the occupant headline is gated only on the **TILE's** fog state (`FogProbe` classifies the GROUND), never on the unit's own visibility, and the occupancy cache it reads indexes all awake units unfiltered (`CustomGridNodeController.cs` `Tick` `:70-100`). A stealth-ambush enemy (`PartUnitStealth` active, not yet spotted — the exact mechanic the game uses for exploration ambushes), a designer-parked `Features.Hidden` NPC, or an `IsInvisible` unit standing on a currently-lit tile is spoken as "<name>, enemy" (plus dead/unconscious state) when the cursor steps onto its tile — while the sighted player sees an empty tile. The leak also applies on NoFow maps (fog-disabled ship decks), where every tile reads "seen". (Nuance: an out-of-combat **Suppressed** NPC is put to sleep and drops out of the occupancy cache — `SleepingUnitsController.cs:65` — so that particular scripted-hiding mechanic does not leak via this path.) Compounds with L1: name the unit, then read its sheet.
- **What the game masks:** `EntityVisibilityForPlayerController.cs:73-108` (view hidden via `Update` → `entity.View.SetVisible(false)` at `:60` — model, overtip and name label all gone; the name overtip is additionally gated on `IsVisibleForPlayer`, `OvertipUnitNameView.cs:69`). The mod's own scanner lens gets this right (`ProxyUnit.cs:46-48`); only this tile-cursor path bypasses it.
- **Fix — gate to add:** In `InteractableDescriber.DescribeTile`, extend the occupant gate to mirror the `ProxyUnit` lens: only speak `node.GetUnit()` when `unit.IsPlayerFaction || unit.IsVisibleForPlayer` (this also covers the NoFow-map case, since `EntityVisibilityForPlayerController` still hides the view there). Keep the existing tile-fog tiering for everything else — a dead lootable corpse remains visible-for-player, so corpse reading is unaffected.

#### L3. Points-of-interest category speaks quest/POI/important map pins the game's local map hides (Hidden flag, unrevealed owner, failed awareness, suppressed entity)
- **The leak:** `RTAccess/Exploration/Scanner.cs:499-515` `MarkerList`: only `LocalMapMarkType.Loot` passes `SafeMarkerVisible` (`:508`); `Poi` / `DestinationMark` / `VeryImportantThing` (`:509-511`) are listed unconditionally, and there is no `entity.Suppressed` filter either. Hidden pins **stay** in `LocalMapModel.Markers` (`SetHidden` does not detach; only `OnDetach`/`OnDeactivate` removes them), so they enumerate freely. A blind player can be told about — and even travel toward (I = `TravelTo`, `Scanner.cs:203,254`) — an "important"/"objective" location the game has deliberately not granted yet.
- **What the game masks:** every sighted local-map pin — not just loot — is hidden whenever its `ILocalMapMarker.IsVisible()` is false: `LocalMapMarkerPCView.cs:42-45` (`SetActive(ViewModel.IsVisible)`) fed per-frame by `LocalMapCommonMarkerVM.cs:15,29-31`; the gates are `LocalMapMarkerPart.cs:65-76` (`!Hidden && Owner.IsRevealed && Owner.IsInGame && IsAwarenessCheckPassed` — quest markers toggled `Hidden` by scripting via `MarkOnLocalMap.SetHidden`, `MarkOnLocalMap.cs:28-38`) and `AddLocalMapMarker.cs:54-64` (`IsInGame && (IsRevealed || ShowIfNotRevealed) && !IsDeadOrUnconscious`), plus the Suppressed-entity filter at `LocalMapVM.cs:96`. Any pin surfaced with `IsVisible() == false` is by definition masked information — the same class of leak the mod already fixed for Loot pins after verifying two undiscovered GoodLoot caches surfaced.
- **Fix — gate to add:** In `Scanner.MarkerList`, apply `SafeMarkerVisible(m)` to **all** marker types (drop the Loot-only special case) and additionally skip markers whose `m.GetEntity()?.Suppressed == true` to match `LocalMapVM.SetMarkers`. **Important:** because hidden pins remain enumerable in `LocalMapModel.Markers`, `ResolveSelected` (`Scanner.cs:574-575`) would still resolve a previously selected pin after it becomes invisible, leaving I = `TravelTo` usable on it — apply the visibility (and Suppressed) gate in `ResolveSelected` or `TravelTo` as well to fully close the leak.

#### L4. Room-map narration reveals the size/class of never-seen rooms (X where-am-I, V exit cycle destination, passive room announcements)
- **The leak:** `RoomMap.Build` (`RTAccess/Exploration/RoomMap.cs:213-551`; the load-bearing fog-free walkability read at `:237-253`) reads raw grid walkability with no fog input, and three spoken surfaces voice its classifications of never-seen space: **X** where-am-I (`Scanner.cs:322-327`) speaks "Room N, <class>" for a planted cursor even on NeverSeen ground — the separate "unexplored" tag at `:332-336` doesn't remove the class word, which is the leak (a sighted player sees only black); **V** exit cycle (`Scanner.cs:345-363`, `Describe(ex.To)` spoken at `:362`) classifies the **destination** room's id + class, which can be wholly unexplored; and the **passive** room-change narration (`RoomMap.cs:170-187` `TickAnnounce`) follows the planted cursor straight through unexplored fog, enumerating room ids/classes as it goes (it is gated on `ExplorationActive` + `HasControl` + the `announce_rooms` setting — none of which is a fog gate). The tile readout itself already models the correct behaviour (NeverSeen tile → only "unexplored").
- **What the game masks:** `LocalMapVM.cs:79-88` composites the fog-of-war mask into the local-map draw (`LocalMapFowScaleOffset`), so never-seen terrain is blacked out; no sighted channel exposes the shape/size classification ("large hall", "stairs", "corridor") or exit structure of unexplored space. For **explored** terrain the geometry-only room map is sighted parity (the map shows explored layout) — only never-seen space leaks.
- **Fix — gate to add:** Gate the class word on fog via `FogProbe.Classify` (`RTAccess/Exploration/FogProbe.cs:62-83`, already keypress-cost; all three call sites are key- or dwell-driven): in `Scanner.WhereAmI` and `RoomMap.TickAnnounce`, when the probe point classifies NeverSeen, speak a class-less variant ("unexplored room" / stay silent for the passive announcer); in `Scanner.CycleExit`, probe `ex.To.Centroid` and degrade the line to "Exit, unexplored" when the destination room is never-seen (a partially explored destination's class is arguably sighted-derivable, which the centroid probe approximates reasonably).

### Low severity

#### L5. Wall tones sonify wall geometry around a cursor parked in never-seen fog
- **The leak:** `RTAccess/Exploration/WallTones.cs:80-108` — `Tick` plays whenever a cursor node resolves (`CursorNode` at `:154-155`); `DistanceToWall` (`:133-144`) raycasts raw grid walkability up to 5 m (~3.7 tiles); no fog gate anywhere in the file. With the tone bed enabled in Continuous mode, a blind player can plant the shared cursor deep in never-seen fog and read the surrounding corridor/room shape from the four cardinal wall tones (loud-left + loud-right = corridor, forward swell = dead-end); in "When moving" mode a stationary cursor fades after the 0.25 s grace window (`:48`, `:90`), so the same layout information is delivered tile-by-tile while **stepping** the cursor through fog. Additionally the 5 m ray passes through never-seen tiles with no per-tile check, so walls inside fog are sonified even from an explored tile near the boundary. Off by default and audio-only — hence low severity — but it is the same "unexplored layout" family as L4, and the mod's own `FogCue` proves the boundary is detectable.
- **What the game masks:** the in-world fog shader samples the same `FogOfWarArea.FogOfWarMapRT` that `FogProbe` reads (never-seen reads `(0,0,·)` — verified live 2026-07-02 per the `FogProbe.cs` doc), and the local map is fog-masked (`LocalMapVM.cs:79-88`) — a sighted player cannot perceive wall layout in unexplored space. Explored-but-fogged tiles are fine (the sighted map shows explored layout).
- **Fix — gate to add:** In `WallTones.Tick`, when the cursor tile classifies `FogProbe.FogState.NeverSeen`, call `FadeOut(dt)` instead of playing — mirror `FogCue`'s pattern (`FogCue.cs:38-66`) of sampling the GPU probe only when the cursor **moves**, rate-capped ~0.1 s, caching the last classification between moves so the per-frame tick never does a readback. Accepted residual: the cursor-tile-only gate closes the parked-in-fog case but the 5 m ray can still read a short distance into fog from an explored tile near the boundary — worth noting in the fix.
